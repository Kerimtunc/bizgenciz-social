# YemekZen QR Menu Elite Edition - Product Requirements Document

## 📋 GENEL BAKIŞ

YemekZen QR Menu Elite Edition, restoranlar ve kafeler için gelişmiş multi-tenant QR kod tabanlı dijital menü sistemi. Bu sistem, geleneksel fiziksel menülerin yerini alarak, müşterilere interaktif, güncel ve zengin içerikli bir deneyim sunar. İşletme sahiplerine ise kapsamlı yönetim paneli, analitik ve operasyonel verimlilik sağlar.

### 🎯 ÇÖZÜLEN PROBLEMLER
- Fiziksel menü maliyetleri ve güncelleme zorlukları
- Müşteri deneyiminin sınırlı olması
- İşletme sahiplerinin menü performansını takip edememesi
- Operasyonel verimliliğin düşük olması
- Multi-tenant platform eksikliği
- B2B/B2C entegrasyon eksikliği
- Gelişmiş analitik ve raporlama eksikliği
- **Süperadmin kontrol eksikliği**: İşletmelerin hangi özellikleri kullanabileceğinin merkezi kontrolü

### 🎯 HEDEF KULLANICILAR
- **Restoran ve Kafe Sahipleri**: Multi-tenant platform üzerinden hizmet alan işletmeler
- **Müşteriler**: QR kod ile menüye erişen ve B2C platformdan sipariş veren tüketiciler
- **İşletme Personeli**: Menü yönetimi, sipariş takibi, mutfak yönetimi yapan çalışanlar
- **Platform Yöneticileri (Süperadmin)**: Platform genelini yöneten, işletmelere özellik atayan ekip

### 1.2 Süperadmin Özellik Atama Sistemi

**Merkezi Kontrol Mimarisi:**
- **Feature Flags Sistemi**: Platform genelinde özellik kontrolü
- **Granular Permission Model**: Görme, kullanma, yönetme seviyeleri
- **Dynamic UI Rendering**: Atama yapılmamış özellikler otomatik gizlenir
- **Real-time Updates**: WebSocket ile anlık özellik güncellemeleri

**Veritabanı Yapısı:**
```sql
-- Özellik tanımları
CREATE TABLE features (
    id INTEGER PRIMARY KEY,
    feature_code VARCHAR(50) UNIQUE NOT NULL,
    name VARCHAR(100) NOT NULL,
    description TEXT,
    category VARCHAR(50) NOT NULL,
    is_active BOOLEAN DEFAULT true,
    dependencies JSON, -- Bağımlılık listesi
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

-- Tenant-özellik atamaları
CREATE TABLE tenant_features (
    id INTEGER PRIMARY KEY,
    tenant_id INTEGER NOT NULL,
    feature_id INTEGER NOT NULL,
    is_visible BOOLEAN DEFAULT false,
    is_usable BOOLEAN DEFAULT false,
    is_manageable BOOLEAN DEFAULT false,
    assigned_by INTEGER NOT NULL, -- Süperadmin user_id
    assigned_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY (tenant_id) REFERENCES tenants(id),
    FOREIGN KEY (feature_id) REFERENCES features(id),
    FOREIGN KEY (assigned_by) REFERENCES users(id)
);

-- Özellik kullanım logları
CREATE TABLE feature_usage_logs (
    id INTEGER PRIMARY KEY,
    tenant_id INTEGER NOT NULL,
    feature_id INTEGER NOT NULL,
    user_id INTEGER NOT NULL,
    action_type VARCHAR(50) NOT NULL, -- 'view', 'use', 'manage'
    timestamp TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY (tenant_id) REFERENCES tenants(id),
    FOREIGN KEY (feature_id) REFERENCES features(id),
    FOREIGN KEY (user_id) REFERENCES users(id)
);

-- Özellik bağımlılıkları
CREATE TABLE feature_dependencies (
    id INTEGER PRIMARY KEY,
    feature_id INTEGER NOT NULL,
    depends_on_feature_id INTEGER NOT NULL,
    dependency_type VARCHAR(20) NOT NULL, -- 'required', 'optional', 'conflicts'
    FOREIGN KEY (feature_id) REFERENCES features(id),
    FOREIGN KEY (depends_on_feature_id) REFERENCES features(id)
);
```

**API Middleware Sistemi:**
```javascript
// Feature Flag Middleware
const featureFlagMiddleware = (requiredFeature, permissionLevel = 'usable') => {
    return async (req, res, next) => {
        const tenantId = req.user.tenant_id;
        const hasPermission = await checkFeaturePermission(tenantId, requiredFeature, permissionLevel);
        
        if (!hasPermission) {
            return res.status(403).json({
                error: 'Feature not available',
                feature: requiredFeature,
                required_permission: permissionLevel
            });
        }
        next();
    };
};

// Dynamic UI Component
const FeatureGate = ({ feature, permission = 'usable', children, fallback = null }) => {
    const { hasPermission } = useFeaturePermission(feature, permission);
    return hasPermission ? children : fallback;
};
```

### 1.3 Veritabanı Mimarisi (Database-First Approach)

**OLTP/OLAP Ayrımı:**
- **Operasyonel Veritabanı (OLTP)**: Günlük işlemler için optimize edilmiş
- **Analitik Veritabanı (OLAP)**: Raporlama ve analiz için optimize edilmiş
- **ETL Pipeline**: Geceleri çalışan otomatik veri dönüştürme süreci

**Multi-Tenant Veri İzolasyonu:**
- Her tabloda `tenant_id` kolonu zorunlu
- Cross-tenant veri erişimi kesinlikle yasak
- Tenant-specific cache ve session yönetimi

**Soft-Delete Politikası:**
- Tüm tablolarda `deleted_at` kolonu
- Veri kaybını önlemek için kalıcı silme yok
- Audit trail için tam geçmiş takibi

### 1.4 API-First Mimari

**Backend Sadece JSON API:**
- Frontend ve backend tamamen ayrık
- Tek API hem web hem mobil için
- RESTful standartlarına uygun endpoint'ler
- JWT tabanlı authentication

**Modern API Standartları:**
- Idempotency desteği (POST/PUT endpoint'leri)
- Asenkron işlem yönetimi (job queue sistemi)
- Global health check endpoint'i
- Swagger/OpenAPI 3.0 dokümantasyonu
